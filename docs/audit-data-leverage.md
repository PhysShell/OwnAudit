# Побочная польза из аудит-данных STS

Статус: размышление / план, не реализация. Фиксирует разбор «у нас 72k находок —
какую ещё пользу из них извлечь». Цифры — из `sts_audit/findings.json` на прогон от
2026-06-25 (72 569 находок).

## 0. Ограничение, которое определяет, что вообще возможно

Код STS **проприетарный и локальный** — наружу можно вынести только агрегаты
(частоты правил, матрицу согласия тулов, распределение тиров). «Датасет на публику»
упирается в то, что без кода он малоинтересен внешним. **Реальная ценность —
внутренняя.** Всё, что включает фрагменты кода, остаётся на машине.

## 1. Ключевая переформулировка: «ложняк» ≠ «не буду чинить»

Два разных зверя, которые все путают:

| | что это | как лечится |
|---|---|---|
| **False positive (FP)** | тул *ошибся*, проблемы нет | чинить правило/конфиг, баг-репорт в тул |
| **True-but-wontfix** | тул *прав*, но не ценно (стиль, микро-опт, легаси-угол) | severity-тюнинг, приоритизация |

Гипотеза: из 72k подавляющая часть «шума» — это **второе**, а не FP. Если смешать
в кучу «ложняки», получишь неверные выводы. Структурно расщепление уже почти видно
(см. §3): `general-quality` 45% — это в основном wontfix-by-nature, а не FP.

### Ловушка «codefix есть ≠ ценно»
Signal density по тулу показывает: roslyn 100% «fixable», codeql/infersharp 100% T3
(detect-only). Но «fixable» = «существует codefix», **не** «стоит чинить». Не путать
покрытие фиксами с ценностью.

### Прочие бесплатные метрики
- **Corroboration rate** = % подтверждённых ≥2 тулами: 3 602 high-confidence из
  15 803 кластеров. (Слабый сигнал шума — тулы смотрят на разное, см. §3.)
- **Signal density по тулу** — кто отрабатывает: own-check 380 находок, но почти все
  ценные; roslyn 62k, но разреженный.
- **Fixability coverage** — % с автоматическим путём фикса ≈ 85%.

## 3. Честно про измерение FP-rate

**Напрямую неизмерим: ground truth нет.** Никто не разметил, что из 72k настоящий
баг. Более того, `suppressed = 0` во всём наборе — даже готовых человеческих вердиктов
(«это не баг») нет, самый дешёвый сигнал отсутствует.

Почему «согласие тулов» — **слабый** сигнал шума: CodeQL и Roslyn проверяют *разные*
вещи, их наборы правил почти не пересекаются (оттого high-confidence всего 3 602 из
72k). Низкое пересечение ≠ «ложняк», это «другая линза». Строить mute-список на
кросс-тул голосовании опасно — заглушить реальный баг хуже, чем оставить шум.

### Строгий способ получить число
Стратифицированная выборка + ручная разметка → экстраполяция с доверительным
интервалом. На выходе: «FP-rate = X% ± Y%» (overall + по правилу/тулу), ранжирование
«какие правила реально шумят», **и размеченный корпус как побочный продукт** (он же
золотой eval для AI-фиксера). Цена — человек размечает выборку; без человека настоящего
числа не бывает.

### Эффективность, которая делает это дешёвым
Находки **однородны внутри правила** — все 5 607 INPC020 практически один шейп. Значит,
**размечать по правилу, а не случайно**: 5–10 примеров на правило → вердикт обобщается
на всё правило. Правила концентрированы → разметив ~30 правил (≈200 находок, но «бьют»
по 55% объёма), характеризуешь больше половины бэклога. Вердикт на правило:
`{сигнал / wontfix / FP-склонное}`. **Это превращает невозможное в вечер работы.**

### Бонус-методология: LLM-as-judge с калибровкой
Локальная модель размечает все 72k (дёшево, но врёт); человек размечает калибровочную
выборку; меряем систематическую ошибку модели и корректируем её разметку → оценка по
всему набору дешевле. Модели не доверяем — она префильтр, истину держит выборка.

## 4. Каталог метрик

| Метрика | Что говорит | Нужно |
|---|---|---|
| Concentration / top-N share | пара mute'ов убирает большую часть шума | данные (есть) |
| Signal density по тулу | кто из тулов отрабатывает | данные (есть) |
| Corroboration rate | доля подтверждённых ≥2 тулами | данные (есть) |
| Fixability coverage | доля с автопутём фикса | данные (есть) |
| **FP-rate** (overall + по правилу, с CI) | сколько тул реально врёт | выборка + разметка |
| **Actionability rate** | доля, ради которой стоит тратить время | выборка + разметка |
| **Cost-to-value по правилу** = объём × (1−actionability) | кого глушить первым | выборка + разметка |
| Debt burn-down / churn | скорость закрытия долга, half-life | ≥2 прогона (база заведена) |

## 5. Варианты побочных продуктов (ранжированы)

1. **Лист «правило → вердикт»** (§2 + §3) — *самое полезное и дешёвое сейчас*. Даёт
   FP/wontfix-число, severity-baseline (`.editorconfig` / `GlobalSuppressions`) и
   размеченный корпус — за одно усилие разметки по правилам.
2. **Behavioral hotspots: churn × findings** — *самое мощное по приоритизации*. Скрестить
   находки с git-историей STS (`git log --numstat`): находки в часто меняемых файлах =
   где баги укусят. Бьёт сырые счётчики в разы (Tornhill, behavioral code analysis).
   ⚠️ Требует git-репо STS — **код STS не в этом репо** (находки ссылаются на внешний
   `Broker/…`, `BrokerDataClasses/…`), делается там, где лежит сам STS.
3. **Майнинг кодмодов** — кластеризовать повторяющиеся `rule+message` (нормализация
   сообщений: выкинуть идентификаторы/числа → «шейпы»), чтобы найти следующую
   высокообъёмную тему для своего applier'а. Пересекается с §5.1 (один движок, два выхода).
4. **Локальный AI-корпус `finding → проверенный патч`** — *преждевременно*. Верифицированных
   патчей почти нет (OWN покрывает 380; off-the-shelf codefix'ы ещё не прогонялись —
   Windows/.NET). Сейчас это построение харнеса сбора, а не датасета. Парк до прогонов
   фиксеров на стенде.
5. **Публичный датасет** — *не советую*. Только агрегаты (код проприетарный), ценность
   внешним низкая.

### Ещё углы (не разобраны)
- **Co-occurrence правил** — какие правила сидят в одном файле вместе («шейпы гниения»).
- **Differential value тулов** — что нашёл тул, что остальные пропустили.
- **Projected burn-down** — что чистит подключение каждого фиксера (план ремедиации с
  прогнозом).

## 6. Рекомендация

- Дёшево и сразу: **лист «правило → вердикт»** (§5.1).
- Мощно по приоритизации: **churn × findings** (§5.2), когда есть STS-репо под рукой.
- «Датасет ради датасета» — мимо; ценность в двух пунктах выше.
